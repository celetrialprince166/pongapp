Under the hood: update-kubeconfig & kubectl get nodes

why a NotReady node still shows up

Two commands, completely different jobs. The first only writes a local file. The second is a real round-trip to the managed control plane — which is exactly why it can list nodes that are too broken to use.

① aws eks update-kubeconfig

LOCAL · one API call
  1. Calls eks:DescribeCluster → gets API URL + CA cert + ARN.
  2. Merges 3 blocks into ~/.kube/config: cluster (URL+CA), user, context (set current).
  3. The user stores no token — an exec plugin: aws eks get-token.
Result: a file on disk. The cluster is barely touched; no node is contacted. Every later kubectl call runs the exec plugin to mint a fresh ~15-min pre-signed STS token.

② kubectl get nodes

ROUND-TRIP · control plane only
  1. Runs the exec plugin → aws eks get-token → bearer token.
  2. GET /api/v1/nodes to the managed endpoint (TLS verified by the CA).
  3. API server: authn (token→IAM via webhook) → authz (RBAC / access-entry) → read etcd.
  4. Returns the Node list; kubectl prints the STATUS column.
Talks to the control-plane endpoint, not the nodes. So it works fine even when every node is broken — which is exactly how we could see them while they were useless.

③ why “NotReady · cni not initialized”

reported by kubelet ON the node
  1. EC2 boots → kubelet starts → registers a Node object → it appears in get nodes.
  2. kubelet readiness needs NetworkReady = a CNI in /etc/cni/net.d.
  3. CNI is dropped by the aws-node (vpc-cni) DaemonSet pod.
  4. vpc-cni never installed → no aws-node pod → no CNI → NetworkReady=false.
Node stays NotReady forever → node-group health check fails after 20 min → CREATE_FAILED. Fix: before_compute=true lands the CNI before nodes join.
file write  →  control-plane query (always works)  ≠  node health (kubelet + CNI on the box)

What was installed & changed

the diff of the session

⬇ Installed

  • Helm v4.2.2winget — winget install Helm.Helm. Not present; needed for the LB Controller chart.
  • AWS Load Balancer Controllerhelm — chart eks/aws-load-balancer-controller into kube-system, bound to the IRSA role. It turns the Ingress into a real ALB.
  • EKS addonsterraform — vpc-cni, kube-proxy, coredns, aws-ebs-csi-driver (the last one standalone).
  • kubeconfig context — aws eks update-kubeconfig wrote the pongapp-prod context.

✎ Changed in the repo

  • modules/eks-cluster/main.tffix — before_compute=true on vpc-cni + kube-proxy.
  • envs/prod-eks/*new root — split out from prod so ECS+EKS coexist.
  • modules/ecr/main.tfguard — prevent_destroy on the shared repos.
  • benchmark/eks/patch-namespace-pss.yamlnew — PodSecurity restricted→baseline.
  • benchmark/eks/patch-backend-svc-healthcheck.yamlnew — ALB HC path /api/health/.
  • benchmark/eks/patch-backend-replicas.yamlnew — backend → 1 replica (RWO EBS).
  • storageclass-gp3-retain.yamlnew — Retain class for Postgres.
